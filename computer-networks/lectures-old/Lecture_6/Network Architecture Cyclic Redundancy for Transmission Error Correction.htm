An internet is a collection of interconnected networks. For the information carrying capacity of the connections, bandwidth, to be shared among all users, a control mechanism must be created so that

the bandwidth can be shared properly.  The Internet's Transmission Control Protocol (TCP) is one of

the most widely used mechanisms for sharing the bandwidth.

 

In contrast to a simple DeMultiplexing protocol like the User Datagram Protocol (UDP), TCP is a more  sophisticated transmission protocol that offers a reliable, connection-oriented, byte-stream service.

The sender writes bytes into a TCP connection and the receiver reads bytes out of the TCP connection. That is, TCP on the source host buffers enough bytes from the sending process to fill a reasonably-sized packet, then sends this packet to its peer on the destination host, as shown in the figure below:

    

TCP (Send Buffer)  -¾¾¾¾®           transmit           ¾¾¾¾® TCP (Receive Buffer)

                                   writes bytes      packet  . .  packet      reads bytes

 

              The packets transmitted between TCP peers in the above figure are called segments,  since each one carries

a segment of the byte stream.  TCP has three mechanisms to trigger the transmission of a segment. First, TCP maintains a variable, Maximum Segment Size (MSS), and it sends a segment as soon as it has collected MSS bytes from the sending process.  Second, TCP transmits a segment when the sending process explicitly asks

it to do so.  Specifically, TCP supports a push operation, and the sending process invokes this operation to effectively flush the buffer of unsent bytes.  (This operation is used in terminal emulators like Telnet, where

each byte has to be sent as soon as it is typed.)  The final trigger for transmitting a segment is a timer that periodically fires; the resulting segment contains as many bytes as are currently buffered for transmission.

  

In this paper, we will examine how transmission errors are detected and corrected by transmission control protocols such as TCP. We will first show how Hamming Codes can be constructed to detect and correct

1-bit transmission errors.  Then, we will evaluate various aspects of  Cyclic Redundancy Check (CRC) codes in comparison to Hamming Codes. It has been said in the literature that in the presence of  burst transmission errors, which each begin and end with a bit error, with zero or more intervening corrupted bits, the Cyclic Redundancy Check codes can be a powerful error detection and correction scheme. 

Its efficiency and the ease with which CRC codes are built and implemented are some of the aspects of

the CRC codes that will be evaluated in this paper.

II.  Cyclic Redundancy Check Codes

 

In the presence of  burst transmission errors, which each begin and end with a bit error, with zero or

more intervening corrupted bits, the Cyclic Redundancy Check (CRC)  can be a useful error detection

and correction scheme.  CRC codes are based upon treating bit strings as representations of polynomials with coefficients of  0's and 1's only.  For example, 110001 has 6 bits and thus represents the six-term generating polynomial 

                                           G(x) =  x5 + x4 + 1

 

Polynomial arithmetic is done modulo 2, according to the rules of algebraic theory. There are no carries for addition, and no borrows for subtraction.  Both addition and subtraction are identical to Å, Exclusive OR.  Long division is carried out the same way as it is in binary except that  the subtraction is done modulo 2.  A divisor is said  "to go into" a dividend if the dividend has as many bits as the divisor.

 

When the CRC code method is employed, the sending TCP and the receiving TCP must agree upon a Generator Polynomial G(x) in advance. Both the high-order and low-order bits of the generator must be 1.

 

The basic idea is to append a checksum to the end of the frame M(x) in such a way that the polynomial W(x) represented by the checksummed frame is divisible by G(X),

i.e., R(x) = Remainder of  W(x) Å G(X) = 0.

 

When the receiver gets the checksummed frame W(x), it divides W(x) by G(x). 

If the remainder of  W(x) Å G(x) = 0, there were no transmission errors.

Otherwise, there must be transmission error(s).

 

Example:  Assume that M(x) =  1 0 1 0 1 1 1  (for 'W')  and  G(x) =  x5 + x4 + 1 =  1  1  0  0  1

III.  Cyclic Redundancy Check Codes in comparison to Hamming Codes

 

The above example shows that there can be transmission errors that the CRC code can not detect.  The goal in creating a CRC code is to select a generating polynomial G(x) that covers the statistically likely errors for a given fault model.  Common generating polynomials that give good error coverage include:

 

CRC-16  =  x16  +  x15   +   x2  +  1

 

CRC-CCITT  =  x16  +  x12   +   x5  +  1

 

CRC-32  =  x32  + x26   +  x23  + x16  +  x12   +   x11  + x10  + x8   +  x7  + x5  +  x4   +   x2  + x + 1

 

On the other hand, the Hamming codes can not only detect but also correct any 1-bit transmission errors.

More research is needed for Hamming Codes that will correct transmission errors of 2 or more bits, even

though the likelihood of such transmission errors are very small (less than 0.003%/32-bit transmission).      

 

The use of  Å (Exclusive OR) in constructing Cyclic Redundancy Check codes (packets) appears to be

computationally more efficient than the complex method by which Hamming codes are built.  But, the

number of check bits (checksums) is exact (no more than needed) in Hamming codes, while they are generally longer and even arbitrary in Cyclic Redundancy Check  codes.
